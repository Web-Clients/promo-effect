PARTEA 1: ARHITECTURĂ BACKEND COMPLETĂ
1.1 FUNDAȚIA BACKEND-ULUI
Filozofia Arhitecturală
Backend-ul va fi construit folosind o arhitectură în trei straturi (three-tier architecture) cu separare clară între prezentare, logică de business și persistență. Această separare permite mentenabilitate, testabilitate și scalabilitate pe termen lung.
Stratul de Prezentare (API Layer):

REST API expus public pentru frontend
GraphQL endpoint pentru query-uri complexe ale dashboardului
WebSocket server pentru notificări real-time
Autentificare JWT pe toate endpoint-urile protejate
Rate limiting per utilizator și per IP
CORS configurat pentru domeniile autorizate
API versioning pentru compatibilitate backwards

Stratul de Business Logic (Service Layer):

Servicii independente pentru fiecare domeniu funcțional
Orchestrare complexă a workflow-urilor
Validare de business rules
Integrare cu servicii externe (Maersk, Terminal49, Gmail)
Queue management pentru task-uri asincrone
Event-driven architecture pentru acțiuni reactive

Stratul de Persistență (Data Layer):

PostgreSQL ca bază de date principală relațională
Redis pentru caching și sesiuni
MongoDB pentru log-uri și date nestructurate
S3-compatible storage pentru documente și fișiere
Database migrations cu version control
Backup automatizat și disaster recovery

1.2 API ENDPOINTS COMPLETE
Filozofia API Design
API-ul va urma principiile REST pentru operațiuni CRUD standard și va expune endpoint-uri GraphQL pentru query-uri complexe de dashboard. Toate răspunsurile vor folosi structuri JSON consistente cu meta-informații despre paginare, sorting și filtrare.
Structura Standard Răspuns:
Toate răspunsurile de succes vor conține:

Campo "success" boolean
Campo "data" cu payload-ul efectiv
Campo "meta" cu informații despre paginare, total records, current page
Campo "timestamp" cu momentul generării răspunsului
Campo "request_id" unic pentru debugging

Toate răspunsurile de eroare vor conține:

Campo "success" false
Campo "error" cu cod eroare și mesaj
Campo "details" cu informații adiționale pentru debugging (doar în development)
Campo "timestamp"
Campo "request_id"

Grupul AUTH (Autentificare și Autorizare)
Endpoint POST /api/v1/auth/register:

Permite înregistrarea unui utilizator nou
Primește: email, password, firstName, lastName, phone, company (opțional), role (doar ADMIN poate seta role)
Validează: email format valid, password strength (minim 8 caractere, o majusculă, o cifră, un caracter special), unicitatea email-ului
Creează user cu status PENDING_VERIFICATION
Generează token de verificare email
Trimite email de confirmare cu link de activare
Returnează: user ID, mesaj că trebuie verificat email-ul
Rate limit: 3 încercări pe IP per oră

Endpoint POST /api/v1/auth/login:

Autentifică utilizatorul cu email și password
Verifică dacă contul este activ și verificat
Compară hash-ul parolei cu bcrypt
Generează JWT token cu expirare 24h
Generează refresh token cu expirare 7 zile
Actualizează last_login și last_login_ip
Returnează: access token, refresh token, user profile complet, permissions
Rate limit: 5 încercări greșite apoi blocare temporară 15 minute

Endpoint POST /api/v1/auth/refresh:

Primește refresh token valid
Verifică validitatea și expirarea
Generează nou access token
Returnează: nou access token cu expirare reîmprospătată

Endpoint POST /api/v1/auth/logout:

Invalidează token-ul curent
Adaugă token în blacklist (Redis)
Returnează: mesaj de succes

Endpoint POST /api/v1/auth/forgot-password:

Primește email-ul utilizatorului
Generează token resetare parolă cu expirare 1 oră
Trimite email cu link resetare
Returnează: mesaj generic (pentru securitate nu dezvăluie dacă email-ul există)
Rate limit: 3 încercări per email per oră

Endpoint POST /api/v1/auth/reset-password:

Primește token resetare și noua parolă
Verifică validitatea și expirarea token-ului
Validează strength parola nouă
Actualizează parola utilizatorului
Invalidează toate session-urile active
Returnează: mesaj succes și redirecționare către login

Endpoint GET /api/v1/auth/verify-email:

Primește token verificare din query string
Verifică validitatea token-ului
Activează contul utilizatorului
Setează status ACTIVE
Returnează: mesaj succes și redirecționare către login

Endpoint POST /api/v1/auth/enable-2fa:

Generează secret TOTP unic
Returnează: QR code pentru scanare în Google Authenticator
Returnează: backup codes pentru recovery

Endpoint POST /api/v1/auth/verify-2fa:

Primește codul din authenticator
Verifică validitatea codului TOTP
Activează 2FA pentru cont dacă valid
Returnează: mesaj confirmare

Endpoint POST /api/v1/auth/disable-2fa:

Necesită parolă curentă pentru confirmare
Dezactivează 2FA
Șterge secret TOTP
Returnează: mesaj confirmare

1.3 BUSINESS LOGIC COMPLEXĂ
Motor de Procesare Email cu AI
Flux Complet Procesare Email:
Când un email ajunge pe adresa configurată de forwarding:

Webhook-ul de la provider (Gmail API, Mailgun) primește notificarea
Sistemul descarcă conținutul complet al emailului (headers, body, attachments)
Identifică limba emailului (română, engleză, chineză) folosind language detection
Extrage textul din attachments PDF dacă există (folosind PDF parsing)
Trimite tot conținutul către OpenAI GPT cu un prompt structurat

Promptul pentru GPT va cere extragerea:

Bill of Lading (B/L) number în orice format
Container number(s) - poate fi unul sau mai multe
Shipping line (Maersk, MSC, CMA CGM, etc.)
Vessel name și voyage number
Port of loading cu cod și nume
Port of discharge
ETD (Estimated Time of Departure)
ETA (Estimated Time of Arrival)
Greutate brută și netă
Tip container (20ft, 40ft, 40HQ, Reefer)
Conținut/commodity
Temperature setting (dacă e reefer)
Orice alte informații relevante

GPT va returna un JSON structurat cu toate datele extrase plus un confidence score pentru fiecare câmp.

Sistemul validează fiecare câmp extras:

Container number față de format standard (4 litere + 7 cifre cu check digit valid)
Shipping line față de lista cunoscută de linii maritime
Port codes față de baza UNLOCODE
Date în format consistent


Caută în baza de date dacă există deja un container cu același B/L sau container number

Dacă există:

Compară datele extrase cu datele existente
Dacă sunt diferențe semnificative, notifică operatorul pentru review
Dacă sunt update-uri minore (ETA ajustat, greutate precizată), actualizează automat
Adaugă emailul ca document atașat la container
Creează tracking event cu tipul "EMAIL_UPDATE"

Dacă nu există:

Creează draft de container nou cu datele extrase
Setează status "PENDING_REVIEW" până când un operator validează
Încearcă să identifice clientul bazat pe: expeditor email, consignee name, istoric
Dacă nu poate identifica clientul cu certitudine, marchează pentru alocare manuală
Trimite notificare către operatori că există container nou de validat
Atașează emailul original ca document


Generează raport de procesare:

Ce date au fost extrase cu succes
Ce date lipsesc și necesită completare manuală
Confidence score general
Actions sugerate



Motor Tracking Automat și Sincronizare
Arhitectura Tracking-ului:
Sistemul va avea multiple surse de date pentru tracking și va trebui să le sincronizeze inteligent:
Surse primare:

API-uri directe de la liniile maritime (Maersk API, MSC API când disponibil)
Terminal49 ca agregator universal pentru toate liniile
Scraping controlat pentru linii care nu au API public
Manual entries de la operatori pentru informații exclusive

Procesul de tracking automat:

La crearea unui container în sistem:

Sistemul înregistrează shipping line și container number
Inițiază subscription la Terminal49 pentru acel container specific
Dacă linia are API direct, înregistrează și acolo pentru redundanță
Programează primul check de tracking în următoarele 5 minute


Verificări periodice automate (Polling):

Fiecare container activ (status != DELIVERED) e verificat la intervale calculate inteligent:
Containere fără ETA: check la fiecare 6 ore
Containere cu ETA > 14 zile: check zilnic
Containere cu ETA între 7-14 zile: check la fiecare 12 ore
Containere cu ETA < 7 zile: check la fiecare 4 ore
Containere întârziate: check la fiecare 2 ore
Containere în port: check la fiecare oră


Procesarea datelor primite de tracking:

Sistemul primește răspunsul API (fie via polling, fie via webhook)
Parsează JSON-ul și extrage toate evenimentele disponibile
Pentru fiecare eveniment nou (care nu există deja în TRACKING_EVENTS):

Validează timestamp-ul
Verifică dacă locația este consistentă cu traseul așteptat
Verifică dacă ordinea evenimentelor are sens logic
Dacă detectează inconsistențe, marchează pentru review manual
Creează tracking event nou în baza de date
Update-ază statusul principal al containerului bazat pe cel mai recent eveniment




Detectare Anomalii și Alerte:

Întârziere detectată: Când ETA actual devine > ETA original + 2 zile

Setează flag "delayed" pe container
Calculează numărul de zile întârziere
Generează notificare automată către client
Notifică echipa internă pentru acțiune
Recalculează impact asupra altor containere ale aceluiași client


Schimbare vessel: Când containerul e transferat pe alt vas

Tracking event special "VESSEL_CHANGE"
Notificare către client cu detaliile noului vas
Recalculare ETA bazată pe noul vessel schedule


Customs hold: Când containerul e reținut de vamă

Flag urgent activat automat
Notificare imediată către client și operatori
Tracking special pentru durata hold-ului


Demurrage risk: Când containerul e în port și se apropie de termenul de free time

Calculează zilele rămase până la demurrage
Notificări escalate: cu 3 zile înainte, cu 1 zi înainte, la începutul demurrage
Inclus estimat cost demurrage pe zi în notificare




Reconciliere între surse multiple:

Când există date din multiple surse (Terminal49 + Maersk API direct)
Sistemul compară datele și folosește algoritmul:

Preferă sursa directă de la linia maritimă (mai actualizată)
Dacă sunt diferențe de ETA, folosește cea mai recentă
Păstrează toate sursele în metadata pentru audit
Marchează inconsistențe majore pentru review manual




Caching inteligent:

Răspunsurile API sunt cached în Redis pentru:

Reducerea numărului de API calls (cost saving)
Viteză mai mare pentru query-uri frecvente
Resilience când API-ul extern e down


TTL pentru cache bazat pe urgența containerului:

Containers cu ETA > 7 zile: cache 4 ore
Containers cu ETA < 7 zile: cache 1 oră
Containers în port: cache 15 minute
